# Implementing an MCP Client

## Overview

The MCP Client is the component that allows an application to communicate with an MCP Server.

In a real project, developers usually build either:

* An MCP Server (exposes tools/resources)
* An MCP Client (consumes tools/resources)

In this course, both are implemented to understand how they work together.

---

# Client Architecture

The client side contains two main components:

## 1. MCP Client

A custom class created by us.

Responsibilities:

* Simplify interaction with the MCP server
* Hide implementation details
* Manage resource cleanup automatically

## 2. Client Session

Provided by the MCP Python SDK.

Responsibilities:

* Maintain the connection to the MCP server
* Send requests
* Receive responses

The session must be cleaned up properly when the application finishes, which is why it is wrapped inside a custom client class.

---

# Why Do We Need the Client?

The client acts as a bridge between:

Application Code ↔ MCP Server

It allows the application to:

1. Discover available tools
2. Execute tools when requested by Claude

Without the client, the application would have no way to interact with the MCP server.

---

# Core Functions

Two essential methods must be implemented.

## list_tools()

Purpose:

Retrieve all tools available on the MCP server.

Code:

```python
async def list_tools(self) -> list[types.Tool]:
    result = await self.session().list_tools()
    return result.tools
```

How it works:

1. Access the active session
2. Call the built-in `list_tools()` method
3. Receive a result object
4. Return the list of tools

---

## call_tool()

Purpose:

Execute a specific tool on the server.

Code:

```python
async def call_tool(
    self,
    tool_name: str,
    tool_input: dict
) -> types.CallToolResult | None:
    return await self.session().call_tool(
        tool_name,
        tool_input
    )
```

How it works:

1. Receive the tool name
2. Receive the tool parameters
3. Send the request to the MCP server
4. Return the result

The tool name and input are typically generated by Claude.

# Complete Application Flow

Run:

```bash
uv run main.py
```

Example question:

"What is the contents of the report.pdf document?"

### Behind the Scenes

Step 1

The application asks the MCP client for available tools.

↓

Step 2

The tool definitions are sent to Claude.

↓

Step 3

Claude determines that `read_doc_contents` is needed.

↓

Step 4

Claude requests execution of that tool.

↓

Step 5

The application uses the MCP client to call the tool.

↓

Step 6

The MCP server executes the tool.

↓

Step 7

The result is returned to the client.

↓

Step 8

The result is sent back to Claude.

↓

Step 9

Claude generates the final response for the user.

---

# Key Takeaways

* The MCP Client communicates with the MCP Server.
* The Client Session handles the actual connection.
* The custom client class simplifies usage and cleanup.
* `list_tools()` discovers available tools.
* `call_tool()` executes tools.
* Claude decides which tools to use.
* The client serves as the bridge between application logic and MCP functionality.

Interview Summary:

"The MCP Client is responsible for connecting an application to an MCP Server. It discovers available tools using `list_tools()` and executes them using `call_tool()`. The client wraps the MCP SDK session to simplify resource management and acts as the communication layer between the application and server-side tools."
